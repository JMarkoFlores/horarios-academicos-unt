# ENTREGA P3 - Motor de Asignación Automática y Ventanas de Atención

## 1. Motor de Asignación Automática

### 1.1 Algoritmo de Generación

El motor de asignación automática sigue el siguiente algoritmo:

```
ALGORITMO GenerarHorarioAutomatico(periodo)
  INPUT: periodo (string)
  
  1. LIMPIAR horarios previos del período con origen = GENERACION_AUTOMATICA
  2. CARGAR entidades:
     - Docentes con sus habilitaciones (DocenteCurso)
     - Cursos con sus requisitos (horas teoría/laboratorio)
     - Ambientes con sus capacidades
     - Grupos
     - Disponibilidades de docentes
     - Parámetros de carga (horas máximas por categoría)
  
  3. ORDENAR docentes por jerarquía:
     - Categoría (PRINCIPAL > ASOCIADO > AUXILIAR > JEFE_PRACTICA)
     - Tipo de contrato (NOMBRADO > CONTRATADO)
     - Fecha de ingreso (antigüedad)
  
  4. PARA CADA curso en cursos:
     a. OBTENER grupo asociado al curso en el período
     b. SI no existe grupo:
        - REGISTRAR conflicto: SIN_DOCENTE
        - CONTINUAR siguiente curso
     
     c. ASIGNAR TEORÍA:
        - MIENTRAS horas_pendientes_teoria > 0:
           i. BUSCAR docente elegible:
              - Tiene habilitación para el curso (tipo TEORÍA)
              - Tiene disponibilidad en el slot
              - No excede carga máxima
              - Orden jerárquico
           
           ii. SI no hay docente elegible:
                - REGISTRAR conflicto: SIN_DOCENTE
                - ROMPER bucle
           
           iii. BUSCAR slot libre:
                - Ambiente compatible con el curso
                - No solapado con otros horarios
                - Dentro de franja institucional
                - Día activo
                - Capacidad suficiente
             
              iv. SI no hay slot libre:
                   - REGISTRAR conflicto: SIN_AMBIENTE
                   - ROMPER bucle
           
           v. CREAR HorarioAsignado:
              - docente_id = docente.id
              - curso_id = curso.id
              - ambiente_id = ambiente.id
              - grupo_id = grupo.id
              - periodo = periodo
              - dia = slot.dia
              - hora_inicio = slot.hora_inicio
              - hora_fin = slot.hora_fin
              - tipo_clase = TEORÍA
              - estado = BORRADOR
              - origen = GENERACION_AUTOMATICA
           
           vi. ACTUALIZAR:
                - horas_pendientes_teoria -= duración
                - horas_asignadas[docente] += duración
                - MARCAR slot como ocupado en memoria
     
     d. ASIGNAR LABORATORIO (mismo proceso que teoría)
  
  5. ANALIZAR conflictos detectados:
     - Cruces de docente
     - Cruces de ambiente
     - Cruces de grupo
     - Carga insuficiente/excedida
  
  6. GUARDAR conflictos en base de datos
  7. RETORNAR resultado con estadísticas
END ALGORITMO
```

### 1.2 Tipos de Conflictos

El sistema detecta los siguientes tipos de conflictos tipificados:

| Tipo de Conflicto | Descripción | Gravedad |
|------------------|-------------|----------|
| SIN_DOCENTE | Horario sin docente asignado | ALTA |
| SIN_AMBIENTE | Horario sin ambiente asignado | ALTA |
| CRUCE_DOCENTE | Docente tiene otro horario en el mismo slot | ALTA |
| CRUCE_AMBIENTE | Ambiente ocupado en el mismo slot | ALTA |
| CRUCE_GRUPO | Grupo tiene otro horario en el mismo slot | ALTA |
| CARGA_INSUFICIENTE | Docente tiene menos horas de las requeridas | MEDIA |
| CARGA_EXCEDIDA | Docente excede su carga máxima | MEDIA |
| SIN_DISPONIBILIDAD | Docente no tiene disponibilidad en el slot | ALTA |
| CAPACIDAD_INSUFICIENTE | Ambiente no tiene capacidad para el grupo | MEDIA |
| DESCANSO_MINIMO | Violación de descanso mínimo entre clases | MEDIA |
| FRANJA_INSTITUCIONAL | Horario fuera de franja institucional | MEDIA |
| DIA_NO_LABORABLE | Horario en día no laborable | MEDIA |

### 1.3 Servicios Involucrados

- **AsignacionService**: Lógica principal de asignación
- **GeneracionAutomaticaService**: Orquesta el proceso de generación
- **DeteccionConflictosService**: Detecta y tipifica conflictos
- **ValidadorHorarioService**: Valida slots individuales

## 2. Sistema de Ventanas de Atención

### 2.1 Flujo de Ventanas

```
FLUJO VentanaAtencion
  1. CREAR ventana:
     - Definir fecha, rango horario, categoría, modalidad
     - Estado: PROGRAMADA
  
  2. INICIAR ventana:
     - CARGAR cola de docentes ordenada por jerarquía
     - Estado: EN_CURSO
     - Generar evento WebSocket: cola_actualizada
  
  3. LLAMAR siguiente docente:
     - Marcar docente como EN_ATENCIÓN
     - Asignar intervalo de tiempo (30 min)
     - Generar evento WebSocket: cola_actualizada
  
  4. SELECCIONAR celda temporal:
     - Docente selecciona slot en grilla
     - Guardar en Redis con expiración 30 min
     - Validar: cruce, disponibilidad, capacidad
     - Generar evento WebSocket: celda_seleccionada
  
  5. CONFIRMAR selección:
     - Verificar carga máxima del docente
     - Crear HorarioAsignado con estado CONFIRMADO
     - Limpiar selección temporal de Redis
     - Registrar en auditoría
     - Generar evento WebSocket: horario_confirmado
  
  6. FINALIZAR ventana:
     - Estado: COMPLETADA
     - Generar evento WebSocket: ventana_completada
END FLUJO
```

### 2.2 Gestión de Selecciones Temporales

Las selecciones temporales se gestionan en Redis con las siguientes claves:

```
seleccion_{ambienteId}_{dia}_{horaInicio}_{periodo}
  - Valor: JSON con datos de selección
  - TTL: 30 minutos (1800 segundos)

selecciones_sesion_{sesionId}
  - Set de todas las selecciones de una sesión
  - Usado para confirmación masiva

selecciones_sesiones_activas
  - Set de todas las sesiones activas
  - Usado para limpieza periódica
```

### 2.3 Validaciones antes de Confirmar

Antes de confirmar una selección, el sistema verifica:

1. **Carga máxima del docente**:
   - Cargar parámetros de carga del período
   - Calcular carga actual del docente
   - Verificar: carga_actual + duración ≤ horas_max_semanal

2. **Validaciones de slot** (ValidadorHorarioService):
   - Franja institucional
   - Disponibilidad del docente
   - Cruces de docente, ambiente, grupo
   - Capacidad del ambiente
   - Descanso mínimo
   - Día laborable

3. **Transacción atómica**:
   - Usar QueryRunner para transacción
   - Si hay error, hacer rollback
   - Limpiar selecciones temporales

### 2.4 Eventos WebSocket

El sistema emite los siguientes eventos en tiempo real:

| Evento | Canal | Datos | Cuándo |
|--------|-------|-------|-------|
| cola_actualizada | ventana_{id} | Estado de cola | Al iniciar, llamar siguiente, marcar ausente |
| celda_seleccionada | ventana_{id} | Datos de celda | Al seleccionar celda temporal |
| celda_liberada | ventana_{id} | Datos de celda | Al deseleccionar celda |
| horario_confirmado | ventana_{id} | Horarios confirmados | Al confirmar selecciones |
| ventana_completada | ventana_{id} | Pendientes | Al finalizar ventana |
| alerta_carga | ventana_{id} | Datos de carga | Al detectar carga excedida |

## 3. Integración con Otros Módulos

### 3.1 Sistema de Auditoría

El sistema de auditoría registra automáticamente:
- Cambios en HorarioAsignado (creación, actualización, eliminación)
- Usuario que realizó el cambio
- Fecha y hora del cambio
- Datos anteriores y nuevos
- IP del usuario
- Motivo del cambio

**Integración automática** en:
- AsignacionService.reasignarManual()
- GestorSeleccionTemporalService.confirmarSelecciones()
- VentanasService (acciones de ventana)

### 3.2 Exportación iCalendar

**Endpoint**: `GET /horarios/docente/:id/ics?periodo=`

**Formato**:
- Estándar iCalendar (RFC 5545)
- Compatible con Google Calendar, Outlook, Apple Calendar
- Incluye recurrencia semanal
- Zona horaria: America/Lima

**Estructura del evento**:
```
BEGIN:VEVENT
UID: {horarioId}@unt.edu.pe
DTSTAMP: {timestamp}
DTSTART;TZID=America/Lima: {fecha}T{horaInicio}
DTEND;TZID=America/Lima: {fecha}T{horaFin}
RRULE:FREQ=WEEKLY;BYDAY={dia};UNTIL={fechaFin}
SUMMARY: {curso.codigo} - {curso.nombre} ({tipo_clase})
DESCRIPTION: {docente, curso, grupo, ambiente}
LOCATION: {ambiente.nombre}
STATUS:CONFIRMED
END:VEVENT
```

### 3.3 Notificaciones Push (Firebase)

**Servicio**: FirebasePushService

**Funcionalidades**:
- Enviar notificación individual a docente
- Enviar notificación masiva a múltiples docentes
- Registrar token de dispositivo
- Eliminar token inválido automáticamente

**Eventos que disparan notificaciones**:
- Cambio de horario
- Turno de selección en ventana
- Confirmación de horario
- Recordatorio 24h antes de ventana

**Configuración requerida**:
- Variable de entorno: `FIREBASE_SERVICE_ACCOUNT_PATH`
- Campo en Docente: `firebase_token`

## 5. Consideraciones Técnicas

### 5.1 Performance

- **Caché**: Redis para selecciones temporales y caché de ambientes
- **Índices**: Índices en campos frecuentemente consultados
- **Transacciones**: QueryRunner para operaciones atómicas
- **Validaciones en paralelo**: Múltiples validaciones simultáneas

### 5.2 Seguridad

- **Roles**: Solo usuarios autorizados pueden generar/reasignar
- **Auditoría**: Todos los cambios quedan registrados
- **Validaciones**: Múltiples capas de validación
- **Expiración**: Selecciones temporales expiran automáticamente

### 5.3 Escalabilidad

- **WebSocket**: Redis pub/sub para múltiples instancias
- **Cola**: Bull para procesamiento asíncrono de notificaciones
- **Base de datos**: PostgreSQL con connection pooling
